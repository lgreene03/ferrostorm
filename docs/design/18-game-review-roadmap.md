# 18. Whole-Game Review and Phased Roadmap

Owner: A16 Games Design Reviewer. Date: 2026-07-14.
Labels: persona:all, gdd:all, phase:2-3, owner:design-review.
Inputs: GDD (doc 02), sim handbook (doc 14), visual style (doc 16), control scheme spec (doc 11), /game client source, /sim source, docs/tickets/phase-1-backlog.md, plus five dimension reviews (playable game loop, UI and HUD, combat feedback and effects, audio, content and modes).

## 1. Verdict

Ferrostorm today is a finished engine wearing an unfinished game. The deterministic simulation is genuinely complete for the Phase 1 scope and verified to the project's own high bar: fixed-point maths throughout, every command verb the GDD needs (move, attack, attack-move, harvest, produce, place, sell, repair, deploy, capture, superweapon launch), pay-as-you-build economy with exact refunds, fog and stealth with honest AI, three skirmish AI personalities, a mission trigger system with three authored missions, binary save and load, a bit-exact replay format, and a lockstep layer whose protocol is soak-tested over loopback to zero desyncs (the protocol is proven; play between two machines remains open, see docs/questions/Q002), all held together by a cross-platform golden-hash CI gate. On top of that sits a handsome but inert 3D replay theatre: twenty baked models, a styled terrain and environment pass in the doc 16 war-room palette, and a camera rig, all driven exclusively by a pre-baked JSON export at a hardcoded path. Between the two layers there is almost nothing. No scene binds a live World into the 3D view, no input issues any order other than a single-unit PathMove in the orphaned 2D debug scene, and there is no HUD, no sidebar, no minimap, no menu, no audio subsystem, no combat effect, and no victory screen anywhere in the client. The player-facing game does not yet exist; what exists is everything required to build it quickly. The decisive observation of this review is that the distance from here to a playable modern RTS is short and almost entirely client-side: Main.cs already proves the live binding pattern in 2D (World, 15 Hz accumulator, SnapshotInterpolator, command batching), ReplayTheater.cs already proves the 3D presentation pattern, and the sim's public surface already answers nearly every question the client needs to ask. Only two genuine sim-adjacent gaps surfaced across all five reviews (a read-only production queue accessor and rally-point support), and both are additive.

## 3. The roadmap

Five phases, ordered by dependency rather than by dimension. The ordering principle is simple: nothing downstream can be evaluated, tuned, or even honestly demonstrated until a human can play a live match in the 3D client, so live gameplay comes first even though it is the least glamorous phase. Each phase lists machine-checkable acceptance criteria first, then the honest list of what requires a human. Throughout every phase two standing gates apply and are already automated: the full sim gate (`dotnet run --project sim/Ferrostorm.Sim.Runner -c Release` exits 0) and the determinism CI including golden hashes must stay green; no phase below is permitted to touch them except where explicitly flagged for Architect sign-off (M3, M5, M25 format option, N14, N16).

### Phase A: Live 3D gameplay

Closes B1, B2, B3, B4 (picking and orders half), B8, M2, M29, N1, N3. This is the keystone phase. Build a BattleController on Battle3D that merges the two proven halves the codebase already contains: Main.cs's live loop (own a World, accumulate real time to 15 Hz, Step with the pending command batch, TakeSnapshot into SnapshotInterpolator, sample at render time) feeding ReplayTheater's actor management (spawn and update keyed on ViewEntity instead of JSON rows, reusing BuildTerrain, BuildEnvironment, and ModelLibrary). Bootstrap a real match the way the runner's export mode already does: load data/maps/skirmish-01.fmap, BuildWorld, SetFaction, GrantCredits, SpawnConstructionYard at the start positions, and drive player 1 with a SkirmishAI whose commands append to the same pending list each tick. Extend ViewEntity in Ferrostorm.Presentation with UnitType, MaxHp, Rank, Carry, BuildProgress, ExplicitTarget, HState, and a Cloaked flag, updating the runner's spectate assertions to match. Implement 3D picking (camera ray to the ground plane), drag-box multi-select, control groups, and the full order surface as context-sensitive right-click (enemy gives Attack, ferrite field with harvester selected gives Harvest, ground gives PathMove) plus A attack-move, S stop, H hold, and shift for Command.Queued. ### Sequencing summary

A before everything because no other phase can be honestly evaluated without live play. B before C because effects and audio hang off the event and command surfaces B exercises, and because a playable-but-silent build is testable while a noisy-but-unplayable one is not. C before D by a soft margin: menus wrapping a mute, effectless battle would demo badly and mislead playtesting, though D's data work (missions, maps, string tables) can proceed in parallel with C at any time since it is sim-and-data-side. E last, strictly. The two sign-off items to file immediately regardless of phase are M3 (QueueContents accessor) and the M7 export fix, since both are small, both unblock later phases, and one needs the Architect's queue.
