## 1. Verdict: why the build reads as basic today

The build is mechanically complete and visually clean, but it reads as an indie prototype rather than a modern RTS because all six analyses converge on the same three root causes. First, the light is flat and sourceless: two identical white directional suns plus a 0.7-energy ambient wash mean nothing is modelled in warm and cool, there is no ambient occlusion or contact shadowing so every unit and structure floats above the ground plane, the fog is configured but invisible, and the frame ends in a pure black void above the map edge. Second, the materials are poor: a single colour-ramped noise texture covers roughly 70 percent of every frame, and every model bake is albedo-only at a flat Metallic 0.25, with no baked AO, no normal maps and no true emissives, so the entire roster reads as painted card under a work lamp, and the resource everyone fights over renders as flat white cones. Third, the world is still and amnesiac: outside a 1.5 second effect window nothing moves (no wheel spin, no recoil, no suspension, no idle motion, projectiles are instantaneous lines) and nothing persists (no scorch marks, no tyre tracks, no wrecks), so battles leave no history and the battlefield reads as a diorama rather than a war. Everything below attacks one of those three causes, and almost all of it is mechanical work a cheaper model can execute against machine-checkable acceptance criteria.

## 2. The levers, ranked

Sixty-nine tickets came back across the six dimensions. Three were duplicates across dimensions and have been deduplicated: one canonical ticket carries the work, and the superseded variants are preserved verbatim in Appendix A with do-not-implement notes, because they contain conflicting values that would corrupt the canonical work if a model implemented both. The remaining sixty-six tickets are grouped into five waves in dependency order. Impact and effort gradings are the analysts' own. Every ticket is marked cheapModelSafe true or false: true means every value is committed and the acceptance criteria are machine-checkable, so a cheaper model may implement and verify it without aesthetic judgement; false means the ticket needs at least one look-and-adjust pass by a stronger model or Luke (implement the committed starting values mechanically, then flag for the Wave 5 taste pass).

Wave summary:

- Wave 1, one afternoon, transformative: the lighting, atmosphere and post-processing switches plus the two grounding quick wins. Mostly S-effort property-block edits in BattlefieldView and project.godot. This is the single biggest read-change per hour of work.
- Wave 2, the motion wave: ANIM-01 through ANIM-11 in order. Rig infrastructure, recoil, wheels, suspension, infantry bob, build-up, deploy, doors, intake, idle motion, death tumble.
- Wave 3, the VFX wave: combat effects density (3a) and camera feel plus UI juice (3b). Projectiles, impacts, scorch, wrecks, damage smoke, superweapon sequence, camera smoothing/zoom/shake, health bars, sidebar feedback, pings, toasts, sound variety.
- Wave 4, the material wave: the re-bake pipeline (4a: EMIT, AO, normals, ORM, resolution, emissive parts, silhouette passes) and the ground-surface pass (4b: heightfield, splat shader, cliffs, shores, scatter, tracks, stains).
- Wave 5, taste and biome: mood presets, biome palettes, the SDFGI experiment, the animated menu backdrop, and the consolidated taste pass over every cheapModelSafe:false flag.

```
shader_type spatial;
uniform sampler2D noise_a : repeat_enable, filter_linear_mipmap;
uniform sampler2D noise_b : repeat_enable, filter_linear_mipmap;
uniform sampler2D detail_normal : hint_normal, repeat_enable, filter_linear_mipmap;
uniform vec3 dust_dark : source_color = vec3(0.085, 0.09, 0.10);
uniform vec3 dust_light : source_color = vec3(0.145, 0.15, 0.165);
uniform vec3 rock_dark : source_color = vec3(0.105, 0.10, 0.095);
uniform vec3 rock_light : source_color = vec3(0.185, 0.175, 0.16);
uniform vec3 gravel_dark : source_color = vec3(0.06, 0.065, 0.075);
uniform vec3 gravel_light : source_color = vec3(0.125, 0.135, 0.15);
varying vec3 wpos;
void vertex() { wpos = (MODEL_MATRIX * vec4(VERTEX, 1.0)).xyz; }
void fragment() {
    float m1 = texture(noise_a, wpos.xz * 0.020).r;
    float m2 = texture(noise_b, wpos.xz * 0.007).r;
    float g1 = texture(noise_a, wpos.xz * 0.11).r;
    float g2 = texture(noise_b, wpos.xz * 0.23).r;
    vec3 dust = mix(dust_dark, dust_light, g1);
    vec3 rock = mix(rock_dark, rock_light, g2);
    vec3 gravel = mix(gravel_dark, gravel_light, 0.5 * (g1 + g2));
    float w_rock = smoothstep(0.45, 0.62, m1);
    float w_gravel = smoothstep(0.55, 0.72, m2) * (1.0 - w_rock);
    vec3 alb = mix(dust, rock, w_rock);
    alb = mix(alb, gravel, w_gravel);
    alb *= mix(0.85, 1.12, texture(noise_b, wpos.xz * 0.003).r);
    ALBEDO = alb;
    ROUGHNESS = 0.96 - 0.05 * w_gravel;
    NORMAL_MAP = texture(detail_normal, wpos.xz * 0.15).rgb;
    NORMAL_MAP_DEPTH = 0.55 + 0.4 * w_rock;
}
```

## 3. Execution rules for the implementing model

These rules are not advisory. They exist because each one encodes a failure this project has already paid for.

1. Golden-hash battery before every commit. `dotnet run --project sim/Ferrostorm.Sim.Runner -c Release` must exit 0 with sim/golden-hashes.txt byte-identical. Every ticket in this document is presentation or art pipeline work: any hash delta is a defect in your change, full stop. Do not rationalise it, revert and find it. Changing a golden hash requires an ADR and Architect sign-off (CLAUDE.md), and nothing here qualifies.
2. Never touch /sim. No float, no System.Random, no engine types there. The client (/game) and art pipeline (/art) are explicitly exempt from the determinism laws: System.Random, FastNoiseLite and Time.GetTicksMsec are legal in presentation code, and several tickets rely on that.
3. Offscreen-capture verification is the proof standard (ledger session 18 pattern). A temporary autoload script drives the scene, asserts node state, and writes the framebuffer to PNG; run Godot with `--audio-driver Dummy` (CoreAudio can hang headless with no visible prompt) and launch windowed runs via `open -a`, never a backgrounded shell (macOS App Nap freezes unfocused render loops). Wipe game/.godot/imported and reimport headless whenever a PackedScene loads null after an asset change, then verify with the 20/20 load-all script rather than trusting a clean import log.
4. Capture before/after diffs as proof. Before starting a wave, capture the three fixed viewpoints (the feel-v2-real.png base overview, the terrain-v2.png bridge view, the slice-v5.png firefight) as the wave baseline. After each ticket, re-capture the viewpoint its acceptance names and byte-compare against the baseline: the acceptance says what must differ and what must not. Store captures under art/proof/wave-N/ with the ticket id in the filename. Session 19 lesson on captures: a stale-path poll once copied the previous run's capture and sent a whole session chasing a phantom regression, so always delete the exact artefact path the app writes before re-capturing, and byte-compare captures before diagnosing anything from them.
5. Never build while a verification Godot instance is running (session 19 lesson). Kill the capture instance, then `dotnet build`, then relaunch. Interleaving them corrupts the build/import state and produces exactly the kind of stale-binary ghost hunt the ledger warns about.
6. One wave per session (sub-waves 3a/3b and 4a/4b each count as a session), one ticket per commit, ticket id in the commit message. If a session is dying mid-wave, commit the completed tickets, run the battery, and write the resume point into the ledger before stopping.
7. Respect each ticket's tuning envelope. Where a spec names a single permitted knob (W1-03 exposure, W1-12 dust alpha, W2-03 wheel sign flip, W4-14 ramp sign) adjust that knob only, once, against a capture. Where a spec names a kill criterion (W1-05 fog budget, W5-03 SDFGI) measure, honour it, and record the numbers in the ledger. Anything beyond that envelope is Wave 5 territory: flag it, do not freelance.
8. Frame budget discipline. Acceptance criteria carry per-ticket frame-time deltas at 1600x900; measure against the wave baseline capture run, and if a wave in aggregate regresses more than 3 ms, stop and report before adding more.
9. Blender work follows doc 17. The committed builder.py already carries the Blender 5.x fixes (join matrix_world, transform_apply defaults, primitive size semantics); re-read doc 17 before touching builder/bake/materials2, bake single models with `blender -b -P art/3d/bake.py -- <name>` while iterating, and remember the .glb.import files pin embedded textures (mode 3), so never hand-delete extracted texture strays.
10. Ledger discipline. Every session appends its results, measurements, kill-criterion decisions and resume point to docs/tickets/phase-1-backlog.md, and every deliverable ends with the standard Changed / Assumed / Needed next footer per CLAUDE.md.

Two things in this roadmap cannot be delegated to any model, cheap or otherwise. The first is the final taste pass, consolidated as W5-05: every cheapModelSafe:false flag in this document funnels into one sitting where Luke and a strong model judge fixed-viewpoint captures and turn single knobs. ### Wave 5 addendum: findings from the post-Wave-4c showcase pass (2026-07-15)

W5-06. Grazing-angle fog wash. A cinematic capture at camera height 5.5
